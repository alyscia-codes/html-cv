# 30-Day Execution Roadmap: Single-Page HTML CV

## Week 1: Foundation — Setup, Structure & Core HTML (Days 1–7)

| Day | Theme | What Gets Done |
| ----- | ------- | ---------------- |
| Day 1 | Environment Setup | Create GitHub account (if needed), set up repo, configure Git locally, and push first commit. |
| Day 2 | HTML Boilerplate | Build the base `index.html` with DOCTYPE, `<html>`, `<head>`, and `<body>` tags — nothing else. |
| Day 3 | Head Section — SEO & Meta | Add all required `<meta>` SEO tags (charset, viewport, description, keywords, author) to `<head>`. |
| Day 4 | Head Section — OG Tags | Add all Open Graph `<meta>` tags (og:title, og:description, og:image, og:url, og:type). |
| Day 5 | Favicon | Create or source a favicon, link it in `<head>`, verify it appears in browser tab. |
| Day 6 | Semantic Layout Skeleton | Add all semantic landmark elements (`<header>`, `<main>`, `<section>`, `<footer>`) with correct IDs and no content yet. |
| Day 7 | Week 1 Audit | Validate HTML via W3C validator, fix all errors, commit clean checkpoint. |

---

## Week 2: Content Build — Every Section Populated (Days 8–14)

| Day | Theme | What Gets Done |
| ----- | ------- | ---------------- |
| Day 8 | Header / Contact Block | Add name, title, address, phone, and email to `<header>` using semantic tags (`<h1>`, `<address>`, `<p>`). |
| Day 9 | Skills Section | Build `<section id="skills">` with heading and skill list matching the reference image layout. |
| Day 10 | Education Section | Build `<section id="education">` with school, degree, dates, and description using `<h3>`, `<p>`, `<time>`. |
| Day 11 | Experience — Job 1 | Add first work experience entry with company, location, title, dates, bullet achievements, and skills line. |
| Day 12 | Experience — Job 2 | Add second work experience entry with identical structure to Day 11. |
| Day 13 | Across the Internet Section | Add `<section id="links">` with LinkedIn and GitHub links using proper `<a>` and `aria-label` attributes. |
| Day 14 | Week 2 Audit | Cross-check every section against reference image, validate HTML, commit. |

---

## Week 3: Polish — Semantics, Accessibility & Quality (Days 15–21)

| Day | Theme | What Gets Done |
| ----- | ------- | ---------------- |
| Day 15 | Semantic Deepdive | Replace any remaining `<div>`/`<span>` with correct semantic tags; audit heading hierarchy (h1→h2→h3). |
| Day 16 | Accessibility Audit | Add `lang` attribute, `aria-label` where needed, ensure all links have descriptive text. |
| Day 17 | README — Setup Instructions | Write `README.md` with project description, how to run locally, and live page URL placeholder. |
| Day 18 | README — Polish | Add screenshot of CV to README, add project requirements checklist, mark all complete. |
| Day 19 | Submission Checklist Review | Verify every item on the official checklist: semantic HTML, SEO tags, OG tags, favicon, single-page layout. |
| Day 20 | Cross-Browser Check | Open in Chrome, Firefox, and Edge — verify structure renders correctly in all three. |
| Day 21 | Week 3 Audit | Final W3C validation pass, fix warnings (not just errors), commit polished checkpoint. |

---

## Week 4: Ship & Learn — GitHub Pages, Submission & Community (Days 22–30)

| Day | Theme | What Gets Done |
| ----- | ------- | ---------------- |
| Day 22 | GitHub Pages Deploy | Enable GitHub Pages from repo settings, verify live URL loads correctly. |
| Day 23 | README — Add Live URL | Update `README.md` with the real GitHub Pages URL, commit and push. |
| Day 24 | OG Tag Verification | Test OG tags using LinkedIn Post Inspector and Facebook Sharing Debugger tools. |
| Day 25 | SEO Verification | Run page through Google Rich Results Test and Lighthouse SEO audit, fix any flags. |
| Day 26 | Community Submission | Submit solution to the project community platform per instructions, include repo URL and live URL. |
| Day 27 | Peer Review — Give Feedback | Review 2–3 other community submissions, leave structured feedback on HTML semantics. |
| Day 28 | Peer Review — Receive & Respond | Read all feedback received on your submission, document changes you agree with. |
| Day 29 | Implement Feedback | Apply agreed-upon improvements from peer review, commit with message "feat: peer review improvements". |
| Day 30 | Final Submission & Retrospective | Confirm all checklist items are live, write a 5-bullet retrospective of what you learned, close the project. |
